    Create src/hooks/useAnimationSequence.ts -- a React hook that controls WHEN pieces become visible. CSS transitions handle the smooth interpolation; this hook only toggles boolean states on a timer schedule.

    **Architecture insight from research:** "React only needs to toggle boolean states, not update positions 60x/sec." The hook returns which pieces are "active" (animated=true) and the board list component applies CSS transition styles. No rAF needed.

    **Exported types:**

    ```typescript
    export type AnimationPhase = 'idle' | 'playing' | 'complete';

    export interface UseAnimationSequenceReturn {
      phase: AnimationPhase;
      activePieceKeys: Set<string>;  // keys of pieces that should be at final position
      wasteVisible: boolean;         // true after all pieces animated
      skipToEnd: () => void;
    }
    ```

    **Hook signature:**
    ```typescript
    export function useAnimationSequence(
      boards: BoardLayout[] | null,
      resultKey: number  // incremented each optimize click
    ): UseAnimationSequenceReturn
    ```

    **Implementation (follows research Pattern 2 + Pattern 4):**

    1. **State:** `phase: AnimationPhase` (useState), `activePieceKeys: Set<string>` (useState), `wasteVisible: boolean` (useState), `timersRef: useRef<number[]>([])` for cleanup.

    2. **Piece key format:** `${piece.pieceId}-${piece.instanceIndex}`

    3. **Animation trigger (useEffect on resultKey + boards):**
       - If boards is null or empty, set phase='idle', clear activePieceKeys, set wasteVisible=false, return
       - Clear all existing timers from timersRef
       - Set phase='playing', activePieceKeys to empty Set, wasteVisible=false
       - Build the stagger schedule:
         a. For each board in order, call sortPiecesForAnimation(board.pieces) to get sorted pieces
         b. Call computeStaggerDelay(sortedPieces.length) to get per-piece delay for this board
         c. For each piece, compute absolute delay = boardStartTime + (pieceIndex * staggerDelay)
         d. boardStartTime for first board = 0 (but use a small initial delay, see two-phase mount below)
         e. boardStartTime for subsequent boards = previous board's last piece delay + 400ms (transition duration) + 200ms (inter-board pause per CONTEXT.md)
       - **Two-phase mount (CRITICAL per research Pitfall 6):** Use `requestAnimationFrame(() => { ... })` before scheduling setTimeout chain. This ensures the browser has painted pieces at their offset positions before triggering transitions.
       - Schedule setTimeout for each piece: `window.setTimeout(() => { setActivePieceKeys(prev => new Set([...prev, pieceKey])) }, absoluteDelay)`
       - Store each timer ID in timersRef.current
       - Schedule final timer after last piece + 400ms transition + 200ms: set wasteVisible=true, then after 500ms more set phase='complete'

    4. **skipToEnd:** Clear all timers. Set activePieceKeys to a Set containing ALL piece keys from all boards. Set wasteVisible=true. Set phase='complete'. This happens synchronously so the CSS `transition: none` trick in Plan 02 can prevent animated transition.

    5. **Cleanup:** useEffect return function clears all timers from timersRef.

    6. **Re-trigger:** When resultKey changes, the useEffect runs again, clearing old timers and starting fresh.

    7. **Stagger timing math for ~2-3 second total:**
       - Per board: computeStaggerDelay returns 30-100ms gap between pieces
       - Per piece CSS transition: 400ms (set in Plan 02's PieceRect)
       - Inter-board pause: 200ms (per CONTEXT.md locked decision)
       - Example: 2 boards x 6 pieces each, stagger=100ms:
         Board 1: pieces at 0, 100, 200, 300, 400, 500ms; last piece done at 500+400=900ms
         Board 2 starts: 900+200=1100ms; pieces at 1100, 1200, 1300, 1400, 1500, 1600ms; done at 2000ms
         Waste fade: 2000+200=2200ms; complete: 2700ms -- fits 2-3s budget

    **Why NOT rAF:** Per research, CSS transitions run on the GPU compositor thread. Using rAF + per-frame setState would run on the main thread and re-render React 60 times/second. The setTimeout approach only triggers ~N state updates total (one per piece), and CSS handles all smooth interpolation. This is dramatically simpler and more performant.
